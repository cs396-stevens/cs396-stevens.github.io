Network Connections

In this lab you’ll be familiarizing yourself with networking and various network CLI tools.

The first tool you’ll be using is netcat or nc.

netcat is a networking tool for most things involving TCP, UDP, and UNIX-domain sockets.

In this series of challenges, the /challenge/run script will create a subnet with a host that you control, and another server, each with their own IP addresses.

You will be dropped into a root shell on the host and be tasked with interacting with the other server in various ways.

Network Scanning

In the Scan challenges, running /challenge/run will put you in a shell on a host in a subnet , where your job is to find the one other server on the subnet that will give you the flag when you connect to it.

The size of the subnet is determined by the subnet mask, (the number after the / in the IP address) which gives the number of top bits in the IPv4 address that correspond to the subnet identifier.

All devices on the same subnet will share that identifier, with the bottom bits providing identification within the subnet.

So if the subnet mask has 24 bits (e.g. 10.0.0.0/24), then there are 8 remaining bits to specify devices on the subnet, giving a maximum of 256 devices that can be connected at once, with IP address from 10.0.0.0 to 10.0.0.255.

Network Monitoring

Network monitoring is incredibly important in practical network security.

The Monitor challenges are opportunities to work with Wireshark, specifically the CLI version of it, tshark.

After running /challenge/run, you will be placed in a shell on a host that is running a client that is regularly receiving the flag data from another server on the network. Your job is to use tshark to capture those packets, read their contents, and recover the flag.

tshark Basics

To start monitoring all incoming and outgoing packets on a network interface (e.g. eth0), you can run the command:

tshark -i eth0

You can use the ifconfig command to list these interfaces and their associated information.

What you will be really interested in, however, is specifying additional conditions for “capturing” packets. tshark has a filter option -f that will allow us to capture specifically packets being received on a particular port. The filter syntax is identical to the Berkeley Packet Filter syntax.

When using tshark, it is useful to first capture some packets using the -w flag option, for which you can specify a filename (.pcap extension is reasonable) for later review with the -r option.

For example:

tshark -f "host 192.168.1.100 and tcp port 8080" -w traffic.pcap

This would capture packets being sent to the indicated host and tcp port, and save the captured packets to a file named traffic.pcap.

We could then review the packets like so:

tshark -r traffic.pcap -Vx

Where -V displays detailed protocol information (from the header) and -x displays the full raw packet bytes in hex with the corresponding ascii character representation.

Other Options

There are plenty of other options for monitoring network traffic. One Python-based option is to use Scapy, a Python library for packet manipulation that is also available on the challenge instance.
